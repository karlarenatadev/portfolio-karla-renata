AWS Certification Simulator

Pipeline de Dados Autônomo com LLM (Google Gemini) e PWA

Python Google Gemini API Pydantic PWA (Vanilla JS) GitHub Actions

crisis_alert O Desafio

Criar uma plataforma de simulados para certificações AWS (CLF-C02, SAA-C03) que se atualizasse de forma autônoma utilizando Inteligência Artificial Generativa. O maior desafio técnico era evitar alucinações da IA — ou seja, garantir que o modelo não inventasse serviços inexistentes ou incluísse tecnologias fora do escopo específico de cada prova.

architecture Arquitetura e Data Quality

Em vez de apenas fazer requisições simples a uma API, construí um Pipeline ETL completo em Python com 4 camadas de Quality Assurance (QA) para os dados gerados pelo Google Gemini:

  • 1. Schema Enforcement (Pydantic): Garante que a IA devolva estritamente um JSON estruturado com pergunta, opções, resposta correta e explicação.
  • 2. Validação Semântica Customizada: Um script (aws_semantic_validator.py) que atua como "juiz de negócios". Ele bloqueia questões que mencionem serviços avançados (ex: Transit Gateway) num simulado de nível fundacional (Cloud Practitioner).
  • 3. Motor de Deduplicação: Utilização do algoritmo SequenceMatcher para medir a similaridade semântica, evitando que a IA popule o banco de dados com perguntas repetidas.
  • 4. Sanity Check: Verificação final de integridade antes do merge no repositório.

Visão do Motor de Validação (Python):

# Exemplo simplificado do fluxo no pipeline_runner.py
def process_new_questions(raw_ai_data):
    # 1. Valida estrutura JSON com Pydantic
    parsed_data = schema_validator.validate(raw_ai_data)

    # 2. Valida semântica (Escopo do Exame)
    if not semantic_validator.is_in_scope(parsed_data, exam_level='CLF-C02'):
        return "Rejected: Out of scope"

    # 3. Detecta Duplicatas contra a base existente
    if duplicate_detector.has_similarity(parsed_data, database):
        return "Rejected: Duplicate"

    return save_to_database(parsed_data)

devices Frontend PWA & Automação CI/CD

O consumo destes dados é feito por uma aplicação frontend desenvolvida em Vanilla JavaScript. A aplicação é um PWA (Progressive Web App) graças ao sw.js (Service Worker), permitindo que os utilizadores façam simulados mesmo offline.

Todo o fluxo de geração e validação de dados roda de forma 100% serverless através do GitHub Actions (verificado nos ficheiros .yml da pasta .github/workflows), mantendo a base de dados de questões sempre atualizada sem intervenção manual.
